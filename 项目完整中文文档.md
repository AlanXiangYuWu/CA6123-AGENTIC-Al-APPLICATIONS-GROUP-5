# CA6123 Group 5 Agentic AI 项目完整中文文档

## 1. 项目概述

本项目是 NTU MCAAI（CA6123）课程的 Agentic AI 小组作业实现，目标是构建一个**多 Agent 自动化产品设计流水线**。  
用户只需输入一句模糊的产品想法，系统即可自动完成从需求澄清、市场调研、PRD、技术架构、代码骨架、QA 审查到最终交付包的全流程生成。

项目核心强调四阶段循环能力：

- **Perceive（感知）**：理解用户输入、检索上下文、读取上游产物
- **Reason（推理）**：各 Agent 基于上下文做结构化决策与生成
- **Action（执行）**：产出工件、执行路由、触发审查/回流
- **Learn（学习）**：通过 QA 回流、多轮迭代、后续评估与微调计划实现持续改进

## 2. 项目目标与价值

### 2.1 业务目标

- 将“产品从想法到可交付设计包”的流程模块化、自动化
- 通过多个职责清晰的 Agent 协作，减少单模型一次性生成的不稳定性
- 为课程作业展示“目标导向、多步骤、自主编排”的 Agentic 能力

### 2.2 教学与演示价值

- 流程路径清晰，便于展示多 Agent 协作机制
- 支持 WebSocket 流式可视化，演示效果直观
- 具备基本 Responsible AI 能力（注入检测、代码安全检查、RBAC）

## 3. 系统总体架构

系统分为三层：

1. **前端层（Vue 3 + Vite）**
   - 输入用户 idea
   - 展示 pipeline 实时进度
   - 展示各阶段 artifacts
   - 下载 final package（JSON）

2. **接口层（FastAPI）**
   - 提供 REST 与 WebSocket API
   - 调用 LangGraph 工作流
   - 在入口处执行注入 guardrail 检查

3. **智能编排层（LangGraph + Gemini + ChromaDB）**
   - 以 `ProjectState` 为共享状态
   - 由 Orchestrator 路由函数决定下一节点
   - 各 Agent 负责本阶段工件生成
   - QA 失败触发回流（最多 2 轮）

## 6. 工作流设计（核心）

### 6.1 Agent 流程

默认主路径：

`customer -> research -> product -> architect -> coder -> qa -> qa_post -> delivery -> end`

### 6.2 Orchestrator 机制

Orchestrator 不是 LLM，而是确定性 Python 路由函数：

- 优先判断是否超过最大回流次数（`revision_round > 2`）
- 按“缺什么补什么”的线性 gating 决定下一步
- QA 不通过时路由回 `coder`
- 全部完成后进入 `delivery`，最后 `__end__`

这种实现有三个优势：

- **确定性**：相同状态必定同样路由
- **可测试性**：路由规则可用普通单元测试验证
- **低成本**：不额外消耗 LLM 调用进行“路由决策”

### 6.3 QA 回流机制

`qa_post` 节点专门处理 QA 失败后的状态重置：

- `revision_round + 1`
- 清空 `code_artifact`
- 清空 `qa_report`

随后回到路由器，让流程自然回到 `coder -> qa`，避免特殊分支逻辑污染。

## 7. 全局状态模型（ProjectState）

共享状态字段包括：

- `messages`：会话消息
- `project_brief`
- `research_report`
- `prd`
- `tech_design`
- `code_artifact`
- `qa_report`
- `final_package`
- `revision_round`
- `citations`
- `guardrail_flags`
- `trace`

设计意义：

- 每个 Agent 只读必要输入、写本阶段产物
- Orchestrator 根据状态字段决定路由
- `trace` 可用于调试与评估路径正确性

## 8. 各 Agent 职责与输入输出

### 8.1 Customer Agent

- 目标：把模糊 idea 转成结构化 `project_brief`
- 输出要求：`<BRIEF>...</BRIEF>` 包裹 JSON
- 兜底策略：若格式不稳定，触发 fallback prompt 强制抽取；仍失败时给默认骨架

### 8.2 Research Agent

- 输入：`project_brief`
- 行为：通过 RAG 检索 business 知识库
- 输出：竞争分析、痛点、机会点、引用列表 `citations`

### 8.3 Product Agent

- 输入：`brief + research_report`
- 输出：PRD（定位、用户流程、优先级功能、非功能需求、成功指标）

### 8.4 Architect Agent

- 输入：`prd`
- 行为：检索 technical 知识库后生成方案
- 输出：技术栈、系统组件、数据模型、API 合同、部署说明

### 8.5 Coder Agent

- 输入：`tech_design`
- 输出：代码骨架（文件列表、入口、运行说明）
- 附加安全检查：对生成代码执行 `code_safety_scan`

### 8.6 QA Agent

- 使用严格模型（temperature=0）增强判定稳定性
- 评估维度：需求覆盖、设计一致性、代码匹配、幻觉/不实风险
- 输出：`passed`、`score`、`issues`、`test_cases`

### 8.7 Delivery Agent

- 聚合所有上游产物
- 输出最终 `final_package`（含 executive summary、部署说明、demo script、roadmap 等）

## 9. LLM 设计与调用策略

`backend/llm/gemini.py` 提供两个客户端：

- `get_llm()`：默认生成（temperature=0.3）
- `get_llm_strict()`：严格判定（temperature=0.0）

环境变量中的 `GOOGLE_API_KEY` 会在运行时注入到进程环境，默认模型来自 `LLM_MODEL`。

## 11. Guardrails（Responsible AI）

### 11.1 注入检测

- 在 `/api/run` 和 `/api/ws/run` 入口调用
- 若 `is_injection=true` 且 `confidence > 0.7`，直接拦截

### 11.2 代码安全扫描

静态规则检测：

- 常见 secret 前缀（如 `AIza`、`sk-`、`AKIA`）
- API key 使用但未通过 `os.environ/getenv`
- `eval()/exec()` 风险调用

命中结果写入 `guardrail_flags`，前端可见。

### 11.3 流程级安全约束

- Agent 本身不直接调用其他 Agent
- 节点跳转只由 Orchestrator + LangGraph 边控制
- QA 回流次数上限防止死循环

## 12. API 设计

### 12.1 REST 接口

- `GET /api/health`：服务状态 + 当前模型
- `POST /api/run`：一次性执行全流程，返回最终状态
- `POST /api/guardrails/injection-check`：单文本注入检测

### 12.2 WebSocket 接口

- `WS /api/ws/run`：流式运行，逐节点推送

事件类型：

- `started`：开始并返回 `thread_id`
- `node_done`：某节点完成，附状态 diff
- `final`：最终状态快照
- `blocked`：被 guardrail 阻断
- `error`：执行异常

## 14. 运行与部署指南

### 14.1 一键启动（推荐）

```bash
./start.sh
./start.sh --no-open
./start.sh --reset
```

启动脚本行为：

- 自动检查 Python/Node/npm
- 自动创建 venv 并安装 Python 依赖
- 自动安装前端依赖
- 启动 FastAPI(:8000) + Vite(:5173)
- 健康检查后可选自动打开浏览器
- Ctrl+C 联动停止并清理子进程

### 14.2 手动启动

后端：

```bash
python -m venv .venv
# 激活 .venv 后
pip install -r requirements.txt
uvicorn backend.main:app --reload --port 8000
```

前端：

```bash
cd frontend
npm install
npm run dev
```

### 14.3 CLI 演示

```bash
python scripts/run_demo.py "I want to build an AI tutoring app for kids"
```

脚本会输出 trace、guardrail 信息、final package 键集合，并保存 `final_package.json`。

## 18. 主要风险与改进方向

### 18.1 风险

- RAG 语料规模不足导致事实支撑有限
- LLM 输出 JSON 仍可能出现边缘格式异常
- QA 判分仍是模型判定，需更强客观评测补充
- 模型版本与配置文档中有历史命名差异（2.0/2.5），需统一

### 18.2 改进建议

- 建立 `index_kb.py` 批量索引真实文档
- 增加 `eval_ragas.py` 与可复现实验数据集
- 增加 `guardrails_test.py` 统计拦截率/误杀率
- 补充关键模块单元测试与集成测试
- 在 README 与 `.env.example` 统一模型默认值

## 19. 建议的验收标准

系统可被认定“可演示可交付”的最低标准：

1. `GET /api/health` 返回 `status=ok`
2. `POST /api/run` 返回完整 `trace` 与 `final_package`
3. 流式运行可看到多次 `node_done` 事件
4. QA 通过时 `final_package` 非空
5. 注入样本能被 guardrail 拦截
6. 代码安全命中可在 `guardrail_flags` 中看到
